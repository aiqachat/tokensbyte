# 系统功能自动化巡检与修复规范 (Skill)

本规范定义了 TokensByte 系统的自动化巡检、故障诊断与修复标准流程。旨在通过严谨的工程实践，确保系统在高可用状态下运行，同时消除日志噪音并保持前端规范性。

## 1. 巡检目标 (Testing Objectives)
- **核心功能验证**：登录、数据加载、财务核算、令牌管理等核心业务流闭环。
- **日志卫生 (Log Hygiene)**：消除无效的 401/404 日志噪音，确保 `logs` 表仅记录有价值的异常。
- **前端规范性 (Frontend Professionalism)**：消除 Ant Design 等 UI 框架的弃用警告，确保控制台整洁。
- **系统稳定性**：及时发现潜在的 Panic 或数据库连接异常。

## 2. 标准操作流程 (Standard Operating Procedure)

### 第一阶段：环境健康度检查
1. **容器状态**：执行 `docker-compose ps` 确保 `frontend`, `backend`, `postgres` 处于 `Up` 状态。
2. **后端可用性**：访问 `/api/health` 验证接口响应（应返回 200 OK，且**不得**在后台产生 401 错误日志）。

### 第二阶段：功能端到端验证
1. **管理员流程**：
    - 登录地址：`/admin0755` (admin/123456)。
    - 验证：用户列表加载、系统配置项修改保存、日志巡回。
2. **普通用户流程**：
    - 登录地址：`/` (chenzs/123456)。 用户不存在时，请创建一个。
    - 验证：钱包余额展示、令牌创建、充值记录加载。

### 第三阶段：异常诊断与修复
- **日志审查**：通过 `SELECT * FROM logs ORDER BY created_at DESC LIMIT 50` 检查最近异常。
- **修复原则**：
    - 优先修复 **Functional Bug**。
    - 其次修复 **Log Spams**（如 favicon 或 health 探测引起的未授权日志）。
    - 最后修复 **Deprecated Warnings**。
- **注意事项**：为了测试而引入 `console.log`、`unwrap()` 等临时代码或验证的增强功能，请务必在测试完成时务必删除。

## 3. 开发严谨性与副作用验证 (Rigor & Integrity)

为了体现开发者的专业性并防止引入新问题，执行巡检与修复时必须遵守以下准则：

### 3.1 批量操作的安全性
- **工具选择**：在 Windows 环境下进行批量替换时，严禁使用 `sed` 等对 UTF-8 编码不敏感的工具。优先使用具有编码感知能力的 IDE 工具或 API 提供的 `replace_file_content`。
- **编码核查**：批量操作后必须随机抽查包含中文、Emoji 的文件，确保无乱码（如 `` 或 `??`）产生。

### 3.2 变更后的闭环验证
- **静态扫描**：每次修复后必须检查 Vite/Webpack 编译控制台，严禁在存在 `PARSE_ERROR` 或 `TypeError` 的情况下交付。
- **冒烟测试**：对每一个修改过的页面，必须使用浏览器实际访问，验证：
    1. 页面是否能够渲染（无白屏）。
    2. 样式是否符合预期（如 Card 的边框消失、Statistics 颜色正确）。
    3. 控制台是否出现新的 React 警告。

### 3.3 功能保全原则
- **严禁破坏原有逻辑**：在进行样式现代化（如 AntD 5.x 升级）时，不得触碰或修改原有的业务逻辑代码（如 API 调用、状态管理），除非是为了修复特定的 Bug。
- **零新增问题**：修复任务的成功定义是“旧问题解决且无新问题引入”。

## 4. 自动化实施指南
当调用此 Skill 时，应遵循以下步骤：
1. 运行巡检子代理（Browser Subagent）模拟用户操作。
2. 检查编译日志与浏览器控制台。
3. 提交修复后，必须通过 **二次闭环验证** 确认页面完整性。
