# 数据库迁移性能优化重构 — 最终实施方案

## 问题根因

[migrations.rs](file:///Volumes/D/aiwwwroot2026/tokensbyte/backend/src/db/migrations.rs) 共 2637 行，每次 Docker 重启**全量执行**。存在三类问题：

1. **性能瓶颈**：百万级 `logs` 表的全表扫描 UPDATE、120 条 ALTER COLUMN TYPE BIGINT 每次重启都重复执行
2. **代码冗余**：同一条 ALTER/UPDATE 语句在文件中出现多次
3. **样板代码堆积**：16 处手写的 `sys_migration_history` 检查模式，每处 8-15 行重复样板

## 改动一：新增 `once_migration!` 宏（架构级提炼）

> [!IMPORTANT]
> 这是核心改进：将一次性迁移的「检查 → 执行 → 记录」三步逻辑封装为统一宏，供新增及存量迁移使用。

**新增位置**：第 33 行后（紧接现有 `exec_ignore!` 宏）

```rust
/// 一次性迁移执行宏：通过 sys_migration_history 确保仅首次部署执行，后续重启自动跳过
/// 用法：once_migration!(pool, "migration_id", "SQL1", "SQL2", ...);
macro_rules! once_migration {
    ($pool:expr, $id:literal, $( $stmt:expr ),+ $(,)?) => {{
        let _m_done: i64 = sqlx::query_scalar(
            concat!("SELECT COUNT(*) FROM sys_migration_history WHERE id = '", $id, "'")
        ).fetch_one($pool).await.unwrap_or(0);
        if _m_done == 0 {
            $( sqlx::query($stmt).execute($pool).await.ok(); )+
            let _ = sqlx::query(
                concat!("INSERT INTO sys_migration_history (id) VALUES ('", $id, "')")
            ).execute($pool).await;
            tracing::info!(concat!("✅ 一次性迁移完成: ", $id));
        }
    }};
}
```

**后续新增一次性迁移只需 3 行**（对比现有 8-15 行样板）：
```rust
once_migration!(pool, "descriptive_migration_id",
    "UPDATE some_table SET ...",
    "ALTER TABLE ...",
);
```

**设计要点**：
- 编译时 `concat!` 拼接，零运行时开销
- 与现有 `exec_ignore!` 风格一致，学习成本为零
- 自动输出完成日志，无需手动 tracing
- 所有 SQL 使用 `.ok()` 忽略错误保持幂等（与现有模式一致），history 记录在最后插入确保失败可重试

### 3.1 logs.action_type 回填（第 720-723 行）

**当前**：4 条 `UPDATE logs SET action_type = ...` 在 exec_ignore! 中无保护，百万级全表扫描

**改为**：从 exec_ignore! 块中提取 4 条 UPDATE，保留索引创建，用 once_migration! 包裹：

## 后续新增迁移规范

> [!TIP]
> 后续开发者新增数据库变更时，遵循以下分类：

| SQL 类型 | 使用方式 | 原因 |
|----------|----------|------|
| `CREATE TABLE IF NOT EXISTS` | 直接 `sqlx::query(...).execute(pool).await?` | PG 原生幂等，轻量 |
| `ALTER TABLE ADD COLUMN IF NOT EXISTS` | `exec_ignore!(pool, ...)` | PG 原生幂等，轻量 |
| `CREATE INDEX IF NOT EXISTS` | `exec_ignore!(pool, ...)` | PG 原生幂等，轻量 |
| `COMMENT ON COLUMN` | `exec_ignore!(pool, ...)` | 无数据操作，轻量 |
| `INSERT ... ON CONFLICT DO NOTHING` | 直接执行 | 已幂等 |
| `UPDATE 大表回填` | `once_migration!(pool, "id", ...)` | **避免重复全表扫描** |
| `ALTER COLUMN TYPE` | `once_migration!(pool, "id", ...)` | **避免重复表检查** |
| `DROP COLUMN` | `once_migration!(pool, "id", ...)` | **不可逆操作** |
| 含 Rust 逻辑的迁移 | 手写 history 检查 | 宏不支持复杂逻辑 |

## 验证计划

### 编译检查
```bash
cd backend && cargo check
```

### 逻辑验证
- 确认所有 `once_migration!` 的 migration ID 全局唯一
- 确认 DDL（ADD COLUMN IF NOT EXISTS）仍在 once_migration! 外部，不受 history 影响
- 确认新旧两种安装场景都正常：
  - **全新安装**：所有 once_migration! 首次执行 → 表为空时 UPDATE 是空操作 → 正确
  - **升级安装**：首次使用新代码时所有 once_migration! 执行一次 → 数据已回填则 WHERE 条件不匹配 → 正确
  - **二次重启**：history 记录已存在 → 全部跳过 → 快速启动

### 性能对比

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 热重启 logs 表全表扫描 | 4-5 次 | **0 次** |
| 热重启 ALTER COLUMN TYPE | ~120 次 | **0 次** |
| 热重启 SQL 执行量 | ~300+ 条 | ~150 条（DDL幂等） |
| 热重启耗时（logs 百万+） | **1-5 分钟+** | **3-10 秒** |
| 冷启动（全新安装） | 不变 | 不变 |

> [!NOTE]
> 所有被 `once_migration!` 保护的 SQL 本身都是幂等的（WHERE 条件约束/IF NOT EXISTS），即使手动删除 `sys_migration_history` 记录重跑也不会产生错误数据或破坏业务功能。
