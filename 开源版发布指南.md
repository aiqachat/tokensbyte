# TokensByte 开源版本 (Open Core) 构建指南

为了保护商业版核心代码，同时方便地维护开源社区版本，系统现在支持通过脚本一键物理隔离并构建开源版代码。

## 1. 原理介绍

在项目根目录下，提供了一个 `build-opensource.sh` 脚本。该脚本基于 **Open Core（开放核心）** 架构设计：

- **前端脱敏**：复制整个项目到 `opensource` 目录后，物理删除非开源商业插件目录（团队营销、素材资产、火山监控/增强、快乐小马等）。前端通过 `import.meta.glob` 动态加载，缺失模块不会导致编译失败。
- **后端脱敏**：修改 `Cargo.toml`，剥离 `commercial_plugins` / `plugin_happyhorse` / `plugin_volcengine_enhance` 等商业 feature；并物理删除对应商业 API/Service 源文件。插件中心通过 `is_plugin_compiled` 过滤展示。
- **数据库**：开源版不删除商业相关表结构与数据；仅插件中心不展示未编译的商业插件。商业版继续可使用全部功能。

### 开源版保留的插件名单（唯一白名单）

| 插件 | name | 说明 |
|------|------|------|
| 创作中心 | `playground` | 用户插件 |
| DocsApi 文档 | `docs_api` | 用户插件 |
| 模型广场管理 | `model_marketplace` | 用户插件 |
| 站点门户 | `site_portal` | 用户插件（feature: `plugin_site_portal`） |
| 站点 Icon 图标库 | `site_icons` | 系统插件（feature: `plugin_site_icons`） |
| 高可用上游渠道 | `high_availability_channel` | 系统内置（核心 relay，无独立前端插件目录） |

## 2. 如何生成开源版代码

每次在商业版主仓库中开发完成并准备发布开源版时，请在**项目根目录**执行：

```bash
chmod +x build-opensource.sh
./build-opensource.sh
```

**执行结果：**
脚本运行完毕后，会在项目根目录生成 `opensource/` 文件夹，内容为已脱敏的纯净代码。

## 3. 本地验证开源版（可选）

**验证前端编译：**
```bash
cd opensource/frontend
npm install
npm run build
```

**验证后端编译：**
```bash
cd opensource/backend
cargo check
# 或 cargo build
```

*(注：剥离商业代码后，Rust 可能出现 `function is never used` 警告，属正常现象。)*

**验证插件中心：**
启动后访问管理端插件列表，应仅看到上述 6 个插件（库中其它插件记录仍可存在，但不会返回给前端）。

## 4. 如何推送到 GitHub

`opensource/` 目录是独立代码库，**不包含**商业版 `.git` 记录。

```bash
cd opensource
git init
git add .
git commit -m "chore: release open source version"
git remote add origin https://github.com/您的用户名/您的开源仓库名称.git
git push -u origin main -f
```

---
> **安全警告：**
> 绝对不要在商业版主工程目录下直接执行 `git push` 到公开仓库。永远只推送由构建脚本生成的 `opensource/` 目录。

## 5. 本地开发与测试工作流

### 核心原则：永远在主目录进行代码编辑
`opensource` 属于**构建产物**。每次执行 `./build-opensource.sh` 都会覆盖该目录。
- **千万不要**直接在 `opensource/` 里改代码。
- 发现 Bug 请回到主仓库修复，再重新运行构建脚本。

### 场景 A：运行【内部商业版】
1. 前端：`cd frontend && npm run dev`
2. 后端：`cd backend && cargo run`

### 场景 B：运行【开源社区版】
1. 根目录运行 `./build-opensource.sh`
2. `cp backend/.env opensource/backend/.env`
3. `cd opensource/backend && cargo run`
4. `cd opensource/frontend && npm install && npm run dev`
